# Backlog: Build CREATE UNIQUE INDEX with Full MVCC Ownership History

## Summary

Make user-table `CREATE UNIQUE INDEX` candidate-complete for every transaction
active across publication. Reconstruct retained unique-key ownership intervals,
preserve the newest mapping in the current unique index, and provide correct
runtime routing to older owners without restricting old transactions from
normal table reads or writes.

## Reference

1. This item is the unique-index portion split from
   `docs/backlogs/000163-create-index-full-mvcc-history.md` during design of
   `docs/tasks/000236-non-unique-create-index-mvcc-candidate-complete.md`.
2. `doradb-storage/src/index/unique_index.rs` stores one MemIndex owner per
   logical key.
3. `doradb-storage/src/index/secondary_index.rs::UniqueSecondaryIndex::lookup`
   checks MemIndex first, so an incorrect historical MemIndex owner can shadow
   the current durable DiskTree owner and cause a false negative.
4. `doradb-storage/src/trx/undo/row.rs::IndexBranch` and
   `doradb-storage/src/trx/row.rs::RowReadAccess::read_row_mvcc` implement the
   current runtime bridge from a newest unique-key owner to an older owner with
   another RowID.
5. `doradb-storage/src/catalog/index.rs` currently validates and builds only
   current unique cold/hot owners. It neither reconstructs historical owner
   intervals nor synthesizes index branches for an index that did not exist
   when those transitions occurred.

## Deferral Context (Optional)

- Defer Reason: Non-unique history can use independent exact
  `(logical_key, row_id)` candidates whose active/delete-masked state is
  resolved by ordinary row MVCC. A unique index has only one physical owner per
  logical key and therefore needs an ordered ownership graph, interval-aware
  validation, atomic branch installation, and a distinct cleanup lifecycle.
  Combining those responsibilities would make task 000236 too broad and would
  obscure the simpler non-unique correctness proof.
- Findings:
  1. The unique MemIndex mapping must continue to select the newest relevant
     owner because MemIndex shadows DiskTree for exact lookup.
  2. Sequential, non-overlapping owners of one key are valid history but cannot
     be represented as multiple unique BTree entries. Older snapshots must
     reach them through runtime `IndexBranch` links.
  3. The existing branches are generated by foreground DML only for indexes
     active at mutation time. CREATE INDEX must reconstruct missing branches
     and attach each one at the correct row-undo transition.
  4. Historical duplicates that overlap in visibility may be unrepresentable
     under the current unique lookup contract. Future design must define
     whether CREATE UNIQUE INDEX rejects such retained history or introduces a
     different representation.
  5. The staged MemIndex can be destroyed on pre-publication failure, but
     synthetic branches mutate live row-undo state. Their installation needs
     rollback ownership or a prevalidated post-commit step under DDL
     exclusion.
  6. Branches and history-only mappings should remain runtime-only because
     pre-crash transactions do not survive restart. Persisting them would
     require a durable cleanup ledger.
  7. Existing undo/index purge may not revisit every branch synthesized by DDL,
     so deterministic reclamation must be designed with explicit ownership.
- Direction Hint:
  1. Keep durable DiskTree contents limited to current unique owners.
  2. Capture one conservative `Global_Min_STS` under DDL exclusion and discard
     only owner intervals whose invalidation CTS is strictly below it.
  3. Group retained versions by encoded logical key, preserve the newest
     mapping, and reconstruct an ordered runtime route to every older owner
     required by a live snapshot.
  4. Define interval-overlap behavior explicitly and test it independently
     from current-state uniqueness validation.
  5. Stage branch intents, prevalidate their row/undo attachment points, and
     make installation, rollback, DROP INDEX, purge, and restart ownership
     explicit.
  6. Re-run the RFC complexity gate during future planning. Do not solve the
     problem by blanket rejection of old-transaction table writes or by a
     separate compatibility index tree without revisiting the tradeoffs
     recorded in task 000236's design discussion.

## Scope Hint

Cover retained hot main-chain versions, committed CDB cold owners, same-RowID
historical keys, different-RowID sequential owners, current-owner selection,
synthetic hot/cold branch targets, interval-aware uniqueness validation,
atomic publication and rollback, normal post-DDL DML, deterministic branch and
mapping cleanup, DROP INDEX interaction, and restart/recovery semantics.

Keep non-unique candidate construction in task 000236, generic streaming or
parallel build optimization, and durable multi-version schema metadata outside
this item.

## Acceptance Hint

For every transaction active across unique-index creation, exact lookup and
range scan through the new index agree with a reference table path for retained
key changes, deletes, and sequential reuse by the same or different RowIDs.
The newest owner remains correct for post-DDL snapshots; old transactions can
continue normal table DML subject only to ordinary MVCC conflicts. Overlapping
historical owners have a documented and tested create-time policy. Synthetic
branches publish atomically, roll back without residue, remain until no active
snapshot can need them, and create no durable cleanup obligation after restart.
